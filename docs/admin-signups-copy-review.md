# Admin Signup Queue Cards Copy Review

| Area                     | Before                                                                                                      | After                                                                                                              | Why it improves review speed / safety                                                               |
| ------------------------ | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------- |
| Status labels            | Mixed chips like `Start here`, `Live trust work`, `Confirmation refresh`, `Approved for publish` in one row | Split into `Workflow state`, `Work type`, and `Priority`                                                           | Operators can scan the same semantic slots on every card instead of decoding mixed chip meaning.    |
| Recommendation label     | `Recommended:` inline in body copy                                                                          | `Recommended action` block with title + supporting reason                                                          | Makes the decision posture the visual center of the card instead of burying it in paragraph text.   |
| Action labels            | Mixed card-specific buttons like `Start review now`, `Details`, `Reject`                                    | Stable action row with `Review`/`Publish`/`Approve claim`, `Request changes`, `View details`, and demoted `Reject` | Keeps action order more predictable and makes destructive behavior less exposed.                    |
| Readiness labels         | `45/100 complete`, `48/100 match-ready` as loose pills                                                      | `Completion`, `Match readiness`, `Blocking`, `Recommended before publish`, `Advisory`                              | Ties readiness numbers to decision meaning and clarifies what actually blocks approval.             |
| Expandable details label | `Review details` / quiet summary row                                                                        | `View submission details` plus explicit `View details` button                                                      | Makes the deeper audit surface discoverable without forcing it to compete with top-level decisions. |
| Timestamp language       | Raw datetime like `4/3/2026, 10:29:41 AM`                                                                   | Operational labels like `Awaiting review 5d`, `In review 2d`, `Submitted Apr 3`, `Updated Apr 18`                  | Helps operators prioritize work faster than raw timestamps.                                         |
| Blocking wording         | Missing issues shown as mixed tags                                                                          | `Blocking` column with explicit “stop publish” framing                                                             | Reduces ambiguity around what is truly unsafe to approve.                                           |
| Advisory wording         | Lower-priority notes mixed into general copy                                                                | `Recommended before publish` and `Advisory` buckets                                                                | Separates “must fix” from “should improve” and lowers false urgency.                                |

## Friction audit

- Inconsistency removed: cards no longer mix workflow, work type, and priority in one undifferentiated chip row.
- Faster scanning: the card now repeats the same order every time: identity, statuses, recommendation, actions, readiness, details.
- Safer decisions: destructive rejection is visually demoted and blocking issues are surfaced before deeper details.
- Assumptions: `reject` remains the only supported destructive removal action for signup cards; no true delete action was introduced because the underlying workflow does not currently expose one.
